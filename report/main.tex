\section{Project Overview}

VegPrice AI is a production-grade, full-stack intelligent system that predicts
next-day vegetable prices for the Chennai wholesale market (Koyambedu and
surrounding mandis). The system combines a traditional machine learning ensemble,
Facebook Prophet time-series models, and NVIDIA NIM large language models for
context-aware price reasoning, all exposed through a REST API and consumed by a
cross-platform Android mobile application.

The project addresses a significant practical problem: vegetable prices in
Chennai fluctuate by 10--40\% day-over-day due to seasonal supply shocks,
festival demand, weather disruptions, and market dynamics. Accurate next-day
price intelligence enables retailers, vendors, and households to make informed
purchasing decisions.

\section{What the Application Does}

A user opens the VegPrice AI Android application and can immediately:

\begin{itemize}[leftmargin=2em, itemsep=4pt]
  \item View tomorrow's predicted price for all 18 major vegetables with trend
        indicators (up/down/stable) and percentage change from today's price.
  \item Scan any vegetable using the phone camera. The app identifies it via
        NVIDIA NIM vision inference and instantly displays today's market price
        and tomorrow's predicted price.
  \item Browse a 7-day price forecast chart with confidence intervals for any
        vegetable.
  \item Compare prices across multiple Chennai markets (Koyambedu, Madhavaram,
        Thiruvallur, etc.) and identify the cheapest source.
  \item Set price alert subscriptions (above or below a threshold) and receive
        Firebase Cloud Messaging push notifications when the threshold is crossed.
  \item View 30-day historical price charts and detailed daily price records.
  \item Star and favourite specific vegetables so they appear at the top of the
        home screen list across app restarts.
  \item Place Android home-screen widgets (three sizes) for live price summaries
        without opening the app.
  \item Toggle between light and dark Material Design 3 themes, persisted across
        sessions.
\end{itemize}

\section{Business Value}

Chennai's Koyambedu wholesale market is one of the largest in Asia, handling
thousands of tonnes of produce daily. Price volatility creates planning
uncertainty across the supply chain. VegPrice AI enables:

\begin{itemize}[leftmargin=2em, itemsep=3pt]
  \item \textbf{Retailers}: Purchase ahead of schedule when a price drop is
        forecast, reducing procurement costs.
  \item \textbf{Vendors}: Optimise buying and selling windows based on predicted
        price direction.
  \item \textbf{Households}: Plan weekly grocery budgets based on predicted
        prices for the coming week.
  \item \textbf{Aggregators}: Monitor market movements across 18 vegetables
        simultaneously via the dashboard endpoint.
\end{itemize}

\section{Data Flow}

The following describes the primary request path from user interaction to
displayed result:

\begin{enumerate}[leftmargin=2em, itemsep=4pt]
  \item \textbf{App launch}: \texttt{HomeScreen} calls \texttt{GET /dashboard}.
        The dashboard handler iterates all 18 vegetables and attempts to fetch
        the latest AI prediction from the Supabase \texttt{predictions} table
        (ordered by \texttt{created\_at DESC}, one row per vegetable). Vegetables
        without a stored AI prediction fall back to the seasonal ML ensemble.

  \item \textbf{AI prediction trigger}: An administrator calls
        \texttt{GET /ai-predict?vegetable=X}. The handler sends the vegetable
        name, current season, weather context, and recent price history to
        NVIDIA NIM. The LLM returns a predicted price, confidence interval,
        trend, and reasoning. The result is immediately persisted to Supabase
        via \texttt{REST INSERT INTO predictions}.

  \item \textbf{Weekly forecast}: \texttt{GET /weekly-forecast} generates a
        7-day array. Day 1 reads the latest AI prediction from Supabase for
        price consistency with the home screen. Days 2--7 are produced by the
        seasonal ensemble model (day-of-year sine/cosine + day-of-week +
        mean-reversion).

  \item \textbf{Camera scan}: The user photographs a vegetable.
        \texttt{ScanScreen} compresses the image to a base64 JPEG and POSTs it
        to \texttt{/scan-image}. The Vercel handler forwards the image to NVIDIA
        llama-3.2-11b-vision-instruct, which returns the vegetable name. The
        handler then checks Supabase for an AI prediction and falls back to the
        seasonal model.

  \item \textbf{Price history}: \texttt{GET /price-history?vegetable=X\&days=30}
        queries Supabase \texttt{price\_records} sorted by date descending and
        returns modal prices for the last N days (maximum 90).
\end{enumerate}

\section{Supabase-First Consistency Model}

A core design decision is the \emph{Supabase-first} prediction strategy. Before
the introduction of this pattern, different screens computed predictions
independently, producing inconsistent prices. The current strategy ensures:

\begin{itemize}[leftmargin=2em, itemsep=3pt]
  \item All four prediction-displaying screens (Home, Forecast, Trends, Scan)
        read from the same \texttt{predictions} table row.
  \item The AI-generated price is the authoritative source; the seasonal ML
        model is the fallback only when no AI prediction exists for a vegetable.
  \item A single admin action (triggering \texttt{/ai-predict}) updates all
        screens simultaneously on the next refresh.
\end{itemize}

\section{Ensemble Strategy}

The ensemble module (\texttt{src/models/ensemble/weighted\_ensemble.py}) combines
predictions from the trained model portfolio:

\begin{enumerate}[leftmargin=2em, itemsep=4pt]
  \item Each model is evaluated on a held-out validation window (last 30 days
        of the training period). RMSE is computed per model per vegetable.
  \item Ensemble weights are computed as the inverse of validation RMSE:
        $w_i = 1/\text{RMSE}_i$, then normalised so $\sum w_i = 1$.
  \item At inference time, the ensemble prediction is the weighted average:
        $\hat{y} = \sum_{i=1}^{N} w_i \hat{y}_i$.
  \item Confidence intervals are propagated: $[\hat{y} - 1.96\hat{\sigma},\,
        \hat{y} + 1.96\hat{\sigma}]$ where $\hat{\sigma}$ is the
        weighted standard deviation of the model predictions.
\end{enumerate}

\section{NVIDIA NIM LLM Prediction}

For AI-mode predictions, the system constructs a prompt with:

\begin{itemize}[leftmargin=2em, itemsep=3pt]
  \item Current vegetable name and market context.
  \item Today's modal price and recent 7-day price trend.
  \item Current Chennai weather conditions (temperature, rainfall, humidity).
  \item Day-of-week and approximate season (Tamil Nadu festival calendar).
  \item An instruction to return a JSON object with \texttt{predicted\_price},
        \texttt{confidence\_lower}, \texttt{confidence\_upper}, \texttt{trend},
        and \texttt{reasoning}.
\end{itemize}

The response is parsed with a regex JSON extractor and validated before being
persisted to Supabase. A cascading model fallback chain (Section \ref{sec:nim_fallback})
ensures availability even when individual models are rate-limited.

\section{Daily Retraining Cycle}

\begin{enumerate}[leftmargin=2em, itemsep=3pt]
  \item \texttt{scripts/daily\_retrain.py} is triggered by macOS launchd or
        the Docker ML worker at 02:00 local time.
  \item New \texttt{price\_records} since the last run are fetched from Supabase.
  \item Features are re-engineered on the updated dataset.
  \item All five model classes are retrained per vegetable.
  \item Best model per vegetable is selected by MAPE on the last-30-day
        validation window.
  \item Updated \texttt{.pkl} artifacts are written to
        \texttt{data/model\_artifacts/\{vegetable\}/\{model\}.pkl} (90 files
        total: 18 vegetables $\times$ 5 models).
  \item Model metrics (RMSE, MAE, MAPE, direction accuracy) are logged to the
        Supabase \texttt{model\_metrics} table for performance tracking.
\end{enumerate}

\section{Overview}

The database is hosted on Supabase (managed PostgreSQL). Supabase provides:

\begin{itemize}[leftmargin=2em, itemsep=3pt]
  \item Auto-generated PostgREST REST API (used by \texttt{api/index.py} for
        direct HTTP queries without a database driver).
  \item Realtime websocket subscriptions for live price updates.
  \item Row-level security (RLS) for user alert data isolation.

\section{Current Implementation}

\begin{notebox}{Authentication Model}
  The Vercel serverless API uses service-level authentication only. The
  \texttt{NVIDIA\_API\_KEY} secures AI calls and the
  \texttt{SUPABASE\_SERVICE\_ROLE\_KEY} authorises database writes. End-user
  API endpoints are currently public (no user authentication required).
\end{notebox}

\begin{itemize}[leftmargin=2em, itemsep=4pt]
  \item \textbf{NVIDIA NIM}: Bearer token (\texttt{Authorization: Bearer
        nvapi-\ldots}) included in every AI request header.
  \item \textbf{Supabase}: Service role key sent in both \texttt{apikey} and
        \texttt{Authorization} headers for all REST calls. The anon key is used
        for read-only public data.
  \item \textbf{CORS}: Currently \texttt{allow\_origins=["*"]} for development
        convenience. Should be restricted to the Vercel deployment origin in
        production.
  \item \textbf{Rate limiting (FastAPI)}: \texttt{slowapi} applies per-IP
        request limiting. Not yet implemented in the Vercel stdlib handler.
  \item \textbf{JWT (FastAPI)}: \texttt{python-jose} with HS256 is configured
        in \texttt{Settings} for future user authentication middleware; not yet
        wired to production routes.
  \item \textbf{Secrets management}: All sensitive values are in \texttt{.env}
        (gitignored) locally and in Vercel environment variables in production.
        They are never committed to source control.
  \item \textbf{Input validation}: Vegetable names are sanitised with
        \texttt{.lower().replace(" ", "\_")}; image payloads are capped at 10 MB;
        alert \texttt{direction} is validated to \texttt{"above" | "below"}.
\end{itemize}

  \item[Mandi]
    Hindi/Tamil term for wholesale agricultural market or trade yard.
  \item[Modal Price]

  \item[Supabase]
    Open-source Firebase alternative built on PostgreSQL, offering a REST API,

    Expo Application Services --- cloud build and over-the-air update platform
    for React Native / Expo applications.

  \item[MAPE]

    Combination of predictions from multiple ML models. This system uses
    inverse-RMSE weighted averaging.

    Day of Year (1--365) --- encoded as sine/cosine for continuous cyclic
    representation of annual price seasonality.
  \item[DOW]
    Day of Week (0--6) --- captures weekly market patterns such as higher

  \item[FCM]
    Firebase Cloud Messaging --- Google's cross-platform push notification
    service used for price alert delivery.
  \item[RLS]
    Row Level Security --- PostgreSQL/Supabase per-row access control policy,
    used to isolate each user's alert data.
  \item[AsyncStorage]
    React Native key-value storage that persists data across app restarts and
    device reboots.
  \item[M3]
    Material Design 3 --- Google's latest design system, used for the
    application's light and dark theming.
  \item[PostgREST]
    Auto-generated REST API over PostgreSQL tables, used by Supabase and
    consumed by \texttt{api/index.py} without a native database driver.
